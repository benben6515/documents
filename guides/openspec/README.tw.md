## 什麼是 OpenSpec？

把 OpenSpec 想像**程式碼的規則**。

不是只說「做這件事」，你用簡單的方式把規則寫下來。

**簡單範例：**

```
不好的程式碼：
「讓使用者登入能動作」

好的規格：
「當使用者有 email 和密碼
 當他們輸入正確資訊
 就讓他們登入」
```

OpenSpec 是輕量級工具。它幫你描述系統應該做什麼。

---

## 為什麼用它？

### 幫你思考

寫規格幫你在寫程式碼**之前**就先思考。

**沒有規格書：**
你可能會忘記東西。你可能建錯東西。

**有規格書：**
你有清楚的計畫。你知道要建什麼。

### 幫你的團隊

當其他人看你的規格書，他們明白：

- 你在建什麼
- 它應該怎麼運作
- 出問題時會怎樣

### 與 AI 工具搭配

OpenSpec 與寫程式代理程式合作：

- Claude
- Cursor
- GitHub Copilot
- 還有很多

你的規格書幫助 AI 理解你要什麼！

## 如何運作

### 簡單字詞

用 3 個簡單字詞：

- **GIVEN（給定）** - 我們有什麼
- **WHEN（當）** - 發生什麼
- **THEN（那麼）** - 應該發生什麼

### 格式

```
GIVEN 某些東西
WHEN 發生某事
THEN 應該發生某事
```

### 每個字詞的意思

**GIVEN（給定）** - 起點
開始之前你有的東西。

想像：「我從哪裡開始？」

**範例：**
```
GIVEN 使用者有帳號
```

---

**WHEN（當）** - 動作
你做什么或什麼發生變化。

想像：「現發生什麼？」

**範例：**
```
WHEN 使用者輸入密碼
```

---

**THEN（那麼）** - 結果
動作發生後應該怎樣。

想像：「結果是什麼？」

**範例：**
```
THEN 使用者應該看到首頁
```

---

### 加入 AND

你可以用 **AND** 來加更多細節。

**用 AND 當：**
- 你需要多個條件
- 你想要多個結果
- 你需要額外資訊

**範例：**

**範例 1：多個條件**
```
GIVEN 使用者已登入
AND 使用者有管理員權限
WHEN 他們點據刪除鈕
THEN 刪除項目
```

**範例 2：多個結果**
```
GIVEN 使用者輸入 email
WHEN 他們點據登入
THEN 顯示他們首頁
AND 發送歡迎訊息
```

**範例 3：複雜情況**
```
GIVEN 使用者有有效的帳號
AND 帳號沒被鎖
AND 使用者輸入正確密碼
WHEN 他們點據登入
THEN 顯示他們儀表板
AND 記住他們的登入 24 小時
```

### 真實範例

**用 AND 登入：**

```
## 場景：使用者勾選「記住我」登入

GIVEN 使用者有有效帳號
WHEN 使用者輸入 email 和密碼
AND 使用者勾選「記住我」選框
THEN 顯示他們首頁
AND 保持他們登入 30 天
```

**用 AND 購物：**

```
## 場景：使用者購買物品

GIVEN 使用者有足夠錢
WHEN 使用者點據「購買」鈕
AND 付款成功
THEN 顯示他們收據
AND 發送確認訊息
AND 從帳戶扣款
```

### 跟你的程式碼在一起

你的規格書留在你的程式碼資料夾。

```
my-app/
├── src/
├── openspec/
│   └── specs/
│       └── auth-login/
│           └── spec.md
└── package.json
```

## 如何撰寫規格

### 步驟 1：選名字

為你的功能選簡單的名字。

```
openspec/specs/
├── user-login/
├── user-logout/
└── change-password/
```

### 步驟 2：寫需求

寫系統應該做什麼。

用簡單的字詞。

```
## 需求：使用者登入

系統必須讓使用者用 email 和密碼登入。
```

### 步驟 3：加入場景

寫 GIVEN-WHEN-THEN 故事。

```
## 場景：成功登入

GIVEN 使用者有有效的帳號
WHEN 他們輸入正確的 email 和密碼
THEN 他們應該看到 App 首頁
```

### 步驟 4：加入邊緣情況

出問題時會怎樣？

```
## 場景：密碼錯誤

GIVEN 使用者有帳號
WHEN 他們輸入錯誤密碼
THEN 他們應該看到「再試一次」
```

# 使用者登入

---

## 目的

讓使用者安全地登入 App。

---

## 需求

### 需求 1

使用者必須能用 email 和密碼登入。

### 需求 2

如果密碼錯誤，系統必須顯示錯誤。

---

## 場景

### 場景 1：成功登入

**GIVEN（給定）** 使用者有有效的 email 和密碼

**WHEN（當）** 使用者輸入正確的 email 和密碼

**THEN（那麼）** 使用者應該看到 App 首頁

---

### 場景 2：密碼錯誤

**GIVEN（給定）** 使用者有有效的 email

**WHEN（當）** 使用者輸入錯誤的密碼

**THEN（那麼）** 系統應該顯示「密碼錯誤」訊息

---

## 邊緣情況

### 邊緣情況 1：沒有帳號

**GIVEN（給定）** 使用者嘗試登入

**WHEN（當）** Email 不在系統中

**THEN（那麼）** 系統應該顯示「找不到帳號」

---

### 邊緣情況 2：帳號被鎖

**GIVEN（給定）** 使用者登入失敗 5 次

**WHEN（當）** 使用者再試一次

**THEN（那麼）** 系統應該顯示「帳號被鎖。請在 10 分鐘後再試」

---

## 小技巧

### 保持簡單

寫得像跟朋友說話一樣。

**不好：**

```
「系統應實作 OAuth2 認證協定」
```

**很好：**

```
「使用者用 email 和密碼登入」
```

### 用清楚的名字

不要用像「feature1」或「thing」這樣的名字。

**不好：**

```
openspec/specs/thing-1/
```

**很好：**

```
openspec/specs/user-login/
openspec/specs/user-logout/
```

### 隨時更新

你的規格書不是刻在石頭上的。

如果你改主意，就更新規格書！

### 跟團隊審閱

在你寫程式碼之前，把你的規格書給團隊看。

問：「這合理嗎？」
